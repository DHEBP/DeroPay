## Money-safety invariants (the hardened spine — these are the non-negotiables)
1. **Single writer of `amount_received`** — only `applyPaymentWithOutbox` writes it, via in-transaction bigint app-sum. The monitor never writes the DB and never forks completion. **This explicitly includes the partial edge:** the engine's old `monitor.on("invoicePartial", … store.updateInvoice(id,{status:'partial', amountReceived}))` path (invoice-engine.ts:537-542) is a SECOND WRITER of `amount_received` sourced from the monitor's stale in-memory `invoice.amountReceived + payment.amount` (monitor.ts:249-250) and is **removed**. The monitor's `invoicePartial` emit (if retained at all) is demoted to **advisory** — it carries NO amount the engine writes; it triggers at most a re-poll, exactly as `invoiceCompleted`/`paymentConfirmed` were demoted in O28/O33. The partial status+amount is set ONLY by the engine's `paymentDetected` handler re-reading store-authoritative totals (invariant 3). *(O3/O15/O16/O24/O34)*
2. **Engine is the SOLE completion decider**, on freshly-committed store totals — the monitor emits only a raw `paymentDetected` per new txid. *(O15)*
3. **Completion AND partial decided on store-authoritative totals, routed through the outbox on EVERY arrival shape** — detection AND confirmation-depth crossing. There is NO separate partial handler that writes: the `partial` transition is produced by the SAME `paymentDetected` handler. On every new txid the engine calls `applyPaymentWithOutbox` (bigint-sums in-tx, reads the committed total back), then `calculateInvoiceStatus` returns `partial` when `0n < total < amount` and `confirming`/`completed` when `total >= amount`; the status update + the matching `invoice.partial`/`invoice.confirming`/`invoice.completed` outbox row are built in ONE tx via `applyInvoiceUpdateWithOutbox`. The monitor's own `invoicePartial` emit is advisory-only (invariant 1) and never supplies the written amount. The `paymentConfirmed` handler re-reads the store-authoritative invoice, runs `calculateInvoiceStatus`, and writes via the same combined tx. Critically this covers the **already-confirmed-at-first-sight** path: `entryToPayment` (monitor.ts:342) stamps `status:'confirmed'` on first detection when `confirmations>=required` (common after restart / low required-confirmations), so that invoice completes via `paymentDetected` and NEVER sees `paymentConfirmed`. If the outbox write lived only in `paymentConfirmed`, that common restart path would mint a `completed` status with **no durable outbox row** (failure 1). Monitor `invoiceCompleted`/`paymentConfirmed` emits stay advisory-only. *(O25/O28/O33)*
4. **Deterministic delivery id for EVERY event type** incl. non-terminal (`confirming`/`partial`), with refire-on-amount-growth and a monotonic-lower-bound contract on amounts. *(O6/O17/O23/O27)*
5. **Status-aware UPSERT**, not bare `INSERT OR IGNORE` — revives + re-signs `dead` rows under the current secret. *(O20)*
6. **Payment-aware expiry** — TTL can't eclipse a funded-but-unconfirmed invoice; a settlement-deadline cap forces a terminal event carrying the partial. *(O19)*
7. **Proven atomicity** — synchronous better-sqlite3 single transaction for status+outbox, with a guard test asserting no `async` slips into the tx body. *(O5/O21)*
8. **Posture wording is honest** — "binds no inbound TCP/UDP listener" (not "zero listeners"); enforced by a runtime-exercised no-listener test PLUS OS-level `lsof`/`ss` in CI (a JS spy can't see native sockets). *(O7/O22)*
9. **`getStats` read-path bigint-safe** at :450 — replace `SUM(CAST(amount_received AS INTEGER))` (typed `number`, then `BigInt(row.total_received||0)` at :470) with a TEXT row-read of `amount_received` + app-side bigint reduce. **This is load-bearing, not optional (O37):** because the constructor sets no `safeIntegers` (:118-130), an i64 SUM > 2^53 arrives ALREADY rounded to a double before `BigInt()` runs — `BigInt(9007199254740993)` silently yields the wrong value, so the bug is NOT only the signed-i64 CAST clamp, it is also the double-return. The TEXT-read + `BigInt(string)` reduce is the canonical fix for ANY numeric bigint-column read and sidesteps the missing-`safeIntegers` defect at root (the new outbox `attempts`/`next_attempt_at` are bounded well below 2^53 so they're safe, but the pattern is: never read a money/precision-critical column as a typed number). *(O4/O26/O37)*
10. **Durable per-invoice scan floor — restart NEVER re-anchors the wallet scan to `currentHeight-5`.** `PaymentMonitor.track()` (monitor.ts:104-116) recomputes `startHeight = max(0, getHeight()-5)` on every call and never persists it; `pollInvoice` passes it as `min_height` (wallet-rpc.ts:215), a HARD lower bound on `GetTransfers`. On restart after >5 blocks of downtime, a payment that landed during downtime falls **below** the re-anchored floor, is never re-scanned, never produces a `paymentDetected`, and the outbox cannot save an event that was never generated (failure 1, and the outbox provably can't reach it). **Fix:** persist a scan floor and anchor `track()` to it instead of `currentHeight-5`. **UNIT CORRECTNESS (O35):** `GetTransfers.min_height` filters on `TransferEntry.height` — BLOCK height (rpc/types.ts:106,122-124; the wallet RPC doc labels it "Minimum height" and entries carry `height` and `topoheight` as distinct fields). On DERO topoheight ≥ block height (they are equal at a single tip — verified on live mainnet `height==topoheight==7223677` and in the doc's example entry `height==topoheight==31297` — and topoheight runs slightly AHEAD only across side-blocks). Feeding a TOPOHEIGHT-derived value into the block-height `min_height` would set the floor too HIGH and could re-drop the very tx. So the floor must be computed in BLOCK-HEIGHT units: `min(createdBlockHeight, min(payment.height)) - REORG_BUFFER`, clamped at 0. `payment.height` is the BLOCK height and is ALREADY persisted on every `Payment` (core/types.ts:40-41, `payments.height` column, round-trips at sqlite.ts:910) — use it, NOT `payment.topoHeight`. `createdBlockHeight` is captured at create time from the daemon's BLOCK height (`getInfo().height`, NOT `daemonRpc.getHeight()` which returns `topoheight` — daemon-rpc.ts:111-113); add a `daemonRpc.getBlockHeight()` reading `getInfo().height`, or read `info.height` directly at createInvoice. No payment can predate `createdBlockHeight`. `track()` derives the floor from the invoice it is handed; no separate cursor column needed beyond `createdBlockHeight`. (REORG_BUFFER additionally absorbs the small topoheight↔height skew, so even an off-by-a-few floor cannot skip a tx.) *(O32/O35)*

## Bottom line
The composition the handoff describes **already exists** as `InvoiceEngine` (wallet+daemon+monitor+store+webhook, with start()-resumes-from-store and event→webhook wiring; its own docstring config is already loopback-in/webhook-out). So this is **not** "build a connector" — it's:
1. Add a **durable webhook outbox** so a missed webhook survives merchant-down / net-blip / **connector restart** (the real hard part).
2. Add a **thin daemon shell + CLI** so the library can run as a long-lived host-side service with **zero inbound listeners**.
3. Close **two real money-safety bugs** an adversarial review found (both verified in source).

Product name: **DeroPay Bridge** — avoids the shipped client-side `WalletConnector` (browser XSWD/WASM) name entirely. CLI: `deropay-bridge`. Export subpath: `dero-pay/bridge`.

## Two verified bugs we must fix (not hypotheticals)
- **Random delivery id** — `createWebhookEvent` mints `randomUUID()` per call (dispatcher.ts:26). A restart-replay would send a *new* id, so the merchant can't dedupe. The handoff's "stable id already exists" is false. **Fix:** deterministic `deliveryId = sha256(invoiceId | eventType | discriminator)`.
- **Bigint truncation** — `SUM(CAST(amount AS INTEGER))` at sqlite.ts:349 and :450 truncates DERO amounts (SQLite `CAST(... AS INTEGER)` is signed i64, AND better-sqlite3 returns the SUM as a JS double with no `safeIntegers` set in the constructor — sqlite.ts:118-130 — so any aggregate above 2^53 atoms is already rounded before `BigInt()` sees it), corrupting `amount_received` that the completed/partial decision reads. **Fix in this PR** (your call): accumulate in bigint-safe app code inside the same transaction.
  - **The fix MUST land in the EXISTING `addPayment` (sqlite.ts:347-351), not only in the new `applyPaymentWithOutbox` (O36).** "Byte-identical default path" is scoped to the **InvoiceEngine** event/webhook emission (so the 182 tests pass), NOT to the store internals. Replacing `addPayment`'s `SUM(CAST(amount AS INTEGER))` with a row-read of `amount` TEXT + app-side bigint reduce + `UPDATE … SET amount_received = <bigint>.toString()` is **output-identical for every input the 182 tests exercise** (verified: `tests/sqlite-store.test.ts:69-100` only sums 200_000n/300_000n/500_000n and asserts the exact bigint total, all far below 2^53), so the regression gate stays green WHILE the truncation is removed on the default (no-sink) path that real non-bridge deployments use. `applyPaymentWithOutbox` reuses the identical bigint-reduce helper. Net: there is NO surviving `SUM(CAST)` decision path, sink or no sink.

## Reliability design (at-least-once, restart-survivable)
One SQLite file = one fsync domain. New additive table `webhook_outbox`:
```
id TEXT PK            -- deterministic deliveryId == X-DeroPay-Delivery; INSERT OR IGNORE collapses dup logical events
event_type TEXT, invoice_id TEXT
payload TEXT          -- frozen signed JSON (bigint→string), value-snapshot taken INSIDE the enqueue tx → stable signature across replays
status TEXT           -- pending | delivering | delivered | dead
attempts INT, next_attempt_at INT(epoch ms)   -- durable backoff (survives restart, unlike setTimeout)
lease_until INT       -- claim lease; reclaims a crash-mid-delivery row instead of deadlocking it
last_error TEXT, created_at INT, delivered_at INT
INDEX(status, next_attempt_at)
```
**Atomic enqueue (closes lost-payment):** new store methods `applyPaymentWithOutbox(...)` / `applyInvoiceUpdateWithOutbox(...)` do the state mutation **and** the outbox INSERT in **one** `db.transaction()` (the store already uses this exact pattern at sqlite.ts:353/:498/:803). The engine builds the event (deterministic id, value-snapshot payload) and passes it *into* the combined call. Crash → both-or-neither on disk.

**Delivery worker loop:** eager-on-boot + interval + kick-on-enqueue.
- CLAIM (atomic UPDATE): rows where `status='pending' OR (status='delivering' AND lease_until<now)` AND `next_attempt_at<=now` → set `delivering`, `lease_until=now+ttl`. The stale-lease clause is what recovers a process killed mid-delivery.
- ATTEMPT: re-sign over the **stored frozen bytes** (identical signature every replay); call a stateless `deliverOnce({url,payload,signature,timeoutMs})` **extracted from** the existing `WebhookDispatcher.attemptDelivery` (reuse HMAC/timeout, don't reinvent).
- ACK 2xx → `delivered`. RESCHEDULE non-2xx → `attempts++`, durable capped-exponential backoff + jitter, `pending`. DEAD after ceiling (default ~24h/50 attempts) → `dead` (never deleted) **+ emit `deadLetter`**, which the bridge wires to a **durable sink** (heartbeat counter + log + nonzero CLI status) — never a dropped in-memory emit.

**Restart re-hydration (both halves):** (A) tracked invoices — `engine.start()` already calls `store.getActiveInvoices()→monitor.track()`, healing the monitor's in-memory `seenTxids`/track-membership amnesia. **But that alone heals only WHO is tracked, not WHERE the scan starts** (O32): `track()` re-anchors `min_height` to `currentHeight-5`, so a payment that landed during downtime is below the floor and never re-scanned. Invariant 10 closes this: `track()` derives a durable scan floor `min(createdBlockHeight, min(payment.height)) - REORG_BUFFER` (BLOCK-height units to match `min_height`'s filter on `TransferEntry.height` — O35) from the persisted invoice. (B) undelivered events — worker's first tick claims pending/stale rows; `delivered` rows excluded → re-deliver undelivered, never replay everything. Net guarantee: after restart, every active invoice is re-tracked AND re-scanned from a floor that cannot skip a downtime payment; once re-detected, completion is decided on store-authoritative totals and written with its outbox row in one tx regardless of whether the tx arrives `confirming` or already-`confirmed` (invariant 3).

**Idempotency contract (documented):** merchant dedupes on `X-DeroPay-Delivery`; at-least-once. DERO corollary: payments are idempotent on-chain by txid.

## Outbound-only posture (the headline property)
- Process opens **zero** inbound listeners. Outbound only: loopback JSON-RPC to wallet (127.0.0.1:10103) + daemon (127.0.0.1:10102), and outbound HTTPS POST to the merchant.
- Health/liveness **without a port**: heartbeat file (atomic tmp+rename) + optional sd_notify + `deropay-bridge status` subcommand (exit 0/1 for Docker HEALTHCHECK). Adding an HTTP `/health` would rebuild the inbound surface we exist to eliminate — explicitly avoided.
- **Enforced as a test:** spy on `http.createServer` / `net.Server.prototype.listen` across the bridge module graph → assert zero calls. Config validation warns if wallet/daemon URLs are non-loopback.

## Residual risks — accepted-with-evidence by the redteam (carry into README/CI)
- **No finite test covers FUTURE transitive deps adding a listener** *(O7/O22)*. Today's dep set binds nothing inbound; keep `lsof`/`ss` in CI and re-run on every dependency bump.
- **Secret rotation mid-flight** *(O20)*. Dead rows self-heal (re-signed under the current secret), but rows still `pending`/`delivering` at rotation were frozen under the old secret and 401 until they hit dead→revive. Ship **drain-before-rotate** guidance in the merchant README.
- **Host clock backward jump** affects `next_attempt_at` selection — it *delays*, never *loses*, delivery. Documented assumption, not a proof; optional monotonic guard.

## Honest correction to the bigint bug claim (from redteam O16)
The bigint truncation at sqlite.ts:349 is **less reachable than first stated**: per-invoice DERO amounts can't exceed supply (~2.1×10¹⁵ atomic, well under 2^63 = 9.2×10¹⁸), so single-payment truncation is economically unreachable on mainnet. Its **real value** is keeping the reconcile-sweep / `getActiveInvoices` re-hydration integer-exact, plus defense-in-depth against dstport-colliding adversarial transfers (not supply-bounded). Still fix :349; reframe the *why*. getStats :450 is analytics-only (O4/O26).
